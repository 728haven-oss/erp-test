# v3 설계서 — 거래처 협업 루프 · 장비별 프로젝트 · 진척 시각화

**작성일:** 2026-07-02 · **전제:** v2(결재·이메일·자재 이력) 위에 추가. 타 ERP 조사(SAP Business Network, Coupa SAN, Oracle iSupplier, Odoo 포털, Anvyl 매직링크, 이카운트 C-Portal, TECHS/rBOM 제번관리, ProShop) 기반.

## 1. 거래처 협업 루프 — 요청하신 5단계의 로직

조사 결론: 요청하신 흐름은 SAP/Coupa급 "공급사 포털"의 축소판이며, **Anvyl의 매직링크**(로그인 없이 토큰 URL로 그 문서만 열람·기입)가 5인 규모에 맞는 구현 모델이다. Odoo 서드파티 앱이 이 패턴을 코드 241줄로 구현한 사례가 있어 가볍게 만들 수 있음이 검증됨.

```
[우리] 견적요청(RFQ) 작성: 품목·수량 (단가 없음)
   → 링크 전달 (테스트: 포털 시뮬레이션 / Flask: 토큰 URL + 이메일)
[거래처] 링크 열람 → 라인별 품번 변경(vendorPart)·단가·납기(일) 기입 → 회신 제출
[우리] 회신 내역 기록·비교 → [발주 전환] (회신 단가·품번 자동 이월, 결재 게이트 적용)
   → 발주서 링크 전달
[거래처] 주문 확인 → "임시 거래명세서" 작성: 준비 수량을 계속 수정 가능 (준비상태 관리)
   → 물건이 전부 준비되면 [확정] → 명세서 잠금 (SAP의 OC→ASN 2단계를 1문서 2상태로 축소)
[우리·긴급] 입고요청(rush) 발송 → [거래처] 준비된 것만 부분 확정 → 부분 출하 (Oracle 분할회신 패턴)
[우리] 입고 처리: 확정 명세서 기준 프리필 + 사진·서류 첨부(GRN 첨부 표준) → 프로젝트 귀속
```

**상태 머신:** RFQ `draft→sent→replied→ordered` / 공급사 명세서 `prep(임시·수정가능)→final(확정·잠금)→received`.

**테스트 버전의 시뮬레이션:** 서버가 없으므로 "공급사 포털" 화면을 같은 앱 안에서 모달로 제공(공급사가 보게 될 화면 그대로). Flask 이행 시 이 화면이 `/p/<추측불가 토큰>` URL이 된다 — 데이터 구조는 동일.

## 2. 장비(호기)별 프로젝트 관리

일본 제번(製番)관리 채택: **수주(프로젝트) → 호기(unit) → 발주/자재/공정** 3계층. 수주 화면에 호기 카드(1호기, 2호기…)를 두고 호기별로 조립 완성도(%)와 도면 참조를 관리. 발주·자재는 v2대로 수주 단위 태깅(호기 단위 태깅은 v4 후보 — 소요 대비 입력 부담 검토 후).

**도면 연결(방향성만):** 파일을 품지 않는다(ProShop·Odoo Documents·Autodesk Vault 하이퍼링크 방식). 호기·품목에 `도면번호 + 도면URL` 필드만 두고 클릭 시 새 탭(구글드라이브/네트워크 폴더). URL이 없으면 도면번호로 드라이브 검색 URL 자동 조립. 리비전 관리(변경 이력+진행 중 호기에 "도면 변경" 배지)는 v4 후보.

## 3. 진척 시각화 (%)

| 지표 | 계산식 | 근거 |
|---|---|---|
| **발주품 입고율** | Σ입고수량 / Σ발주수량 (그 수주에 연결된 발주 전체) | SAP ETO·Genius의 procurement progress |
| **자재 사용률** | Σ사용(조립 투입) / Σ프로젝트 귀속 자재(직납 입고 + 공용 불출) | kit completeness 관행 |
| **조립 완성도** | 호기별 % 직접 입력(담당자 체크) 의 평균 | 소기업 표준 = 셀프 리포트 |
| **프로젝트 완성율** | (사용률 + 조립 완성도) / 2 — 입고율은 별도 표시 | 단일 표준식 없음(제품마다 상이) → 요청 정의대로 "사용률+조립완성도" 합성. 가중치 조정은 설정으로 확장 가능 |

수주 화면에 4개 진행바, 대시보드 제작중 목록에 완성율 미니바. "자재 사용(투입)" 처리 화면에서 프로젝트 귀속 자재의 가용 잔량 대비 투입 수량을 기록(사용률의 원천 데이터).

## 4. 데이터 추가 (Flask 스키마 반영용)

`rfqs`(+lines: vendor_part, price, lead_days), `po.supplier_note`{status prep/final, lines ready qty}, `po.rush`{date,note}, `trans.attachments`(파일명/이미지 — Flask에서는 파일 저장+경로), `orders.units`(name, progress, drawing_no, drawing_url), `uses`(order_id, item, qty — 조립 투입), 품목 `drawing_no/drawing_url`. 포털 토큰 테이블(Flask): `portal_tokens`(doc_type, doc_id, token, expires).

## 5. v3에서 구현하지 않는 것 (경계)

공급사 로그인 계정(토큰 링크로 대체), 도면 파일 저장·리비전 자동 플래그, 호기 단위 발주 태깅, 이나즈마선 간트, ASN 운송정보, 3-way 매칭 자동화 — 전부 v4+ 후보로 문서화만.

## v3 추가분 (2차) — 일정·계정·본사 설정·공급사 모드

**조사 근거:** 이카운트 일정관리(월간 뷰·유형·공유·"입출고 예정일을 달력에"), kintone("전표의 날짜 필드를 달력으로 렌더링" — 납기·수금·입고예정을 이중 입력 없이 자동 표시, 본 구현의 핵심 모델), 사이보우즈 Office(일정 유형별 색상 프리셋), Odoo Activities(문서에 기한 달린 일정)·Milestones(기한 초과=빨강), Monday Deadline Mode+한국 D-day 관행, OWASP(계정·비밀번호 표준), Odoo 내부/포털 사용자 2계층, SAP Ariba Standard(무료 경량 계정).

1. **일정(달력)**: 월간 달력 + 레이어 체크(일정/납기/수금/입고예정 — 전표에서 자동 렌더링, 저장 안 함) + 담당자 필터 + 유형 색상 5종 + 날짜 클릭 상세/추가. 데드라인 지정 시 D-day 배지(D-7 주황, 초과 빨강), 대시보드에 최근접 데드라인 카드, 수주 상세에 프로젝트 일정 카드.
2. **계정(ID/PW/이메일)**: 사용자마다 loginId·비밀번호·이메일 동반. 로그인 모달(테스트 계정 kim/1234, park/1234). 설정>사용자 관리(추가·역할·비번 재설정·비활성, 본인 보호 가드). **비용 경계**: 테스트=평문 비교 / 실개발 준비완료=Argon2 해시+세션+이메일 재설정 토큰(30분·1회용, 무료 SMTP) / SMS 2FA·SSO는 유료라 제외(자리만).
3. **본사 설정(#/settings)**: 회사 정보(상호·사업자번호·대표·업태), 사업자등록증 첨부(이미지 미리보기), 공인인증서 등록(명칭·만료 D-day — 파일 업로드·서명은 Flask 단계, 홈택스 중계 API는 유료), 홈택스 연계 토글(시뮬레이션·send_to_hometax 자리), 통장 연계(은행·계좌·자동수집 토글 — 오픈뱅킹 자리, 현재 CSV로 동일 흐름), SMTP 자리.
4. **공급사 모드(#/vendor)**: 공급사 입장에서 전 과정 테스트 — ①받은 견적요청→견적 작성(품번·단가·납기) ②받은 발주서→**납기 회신(출하 예정일)**+임시 명세서 준비수량 갱신→전량/부분 확정 ③납품 이력. 발주 화면에 공급사 회신 납기 표시.
5. **통장 내역에 은행 표기**: 계좌(은행)별 구분 + CSV 업로드 시 은행 지정, 중복 감지 키에 은행 포함.
6. **거래처 담당자**: 등록/수정 화면에서도 담당자 추가·수정·삭제(즉시 저장).

**양사 상호 거래의 실개발 구조 (질문 답변):** 테스트 단계는 한 화면에서 역할 전환으로 시뮬레이션하지만, 실개발에서는 서버(Flask)가 중계자가 된다 — ①우리 ERP가 문서별 추측불가 토큰 URL 생성 → 공급사에 메일/알림톡 발송 ②공급사는 로그인 없이(또는 무료 경량 계정, Ariba Standard 모델) 그 URL에서 회신·명세서 작성 ③회신 데이터는 같은 DB에 기록되어 우리 화면에 즉시 반영. 공급사도 같은 프로그램을 쓰는 경우 Monitor ERP의 M2M(메일에 PDF+XML 동봉 → 수신측 자동 등록) 패턴으로 프로그램 간 자동 거래까지 확장 가능.

## v3 추가분 (3차) — 문서 발송·검색·장부 분리·모바일

**조사 근거:** 이카운트([저장/전표] 즉시 거래명세서 출력, Search 패널의 조건검색), 얼마에요(거래명세표 양식 항목), 홈택스 세금계산서 표준 서식, 이카운트 자금현황표 vs 매출·매입장(현금주의/발생주의 장부 분리), 경리나라 자금일보.

1. **영업 문서 발송**: 거래명세서·세금계산서 [양식/인쇄] — 표준 서식(공급자란은 설정>본사 정보 자동 반영) → 인쇄/PDF 저장 → 메일 첨부(무료 경로). SMTP 연동 시 자동 첨부(자리). 거래명세서 **직접 작성**(수주 없는 단발 판매) 추가.
2. **조건 검색·필터**: 품목(브랜드 셀렉트) + 견적/수주/납품/계산서/발주/입고 전 목록에 공통 필터바 — 키워드(번호·품목·규격·거래처)·업체별·브랜드별·기간.
3. **장부 통합·분리 조회**: 자금/통장대사를 "장부" 메뉴로 통합. 💰현금 장부(통장 기준 실제 돈: 입출금·누적잔액·월별 흐름) vs 🧾전산 장부(세금계산서 기준 발생: 월별 매출/매입, 부가세 대사용) 분리 — **두 장부의 차이 = 미수금·미지급금**이라는 원칙을 화면에 명시.
4. **모바일 대응**: 반응형 CSS — 좁은 화면에서 사이드바가 상단 가로 메뉴로, 표는 가로 스크롤, 달력·카드 축소, iOS 입력 확대 방지. (추후 앱/모바일 뷰의 기반)

## 원가 확정 원칙 (지시 반영 — 불변 규칙)

**"구매 원가는 매입 세금계산서 발행 이후 변동 불가."** 시스템 보장 구조: ①매입 원가는 입고 라인에 시점 스냅샷으로 저장(품목 단가 변경 무영향) ②매입계산서 수취 시 입고 건 수정·삭제 잠금, 정정은 수정세금계산서 경로만 ③불출 단가·공수 시급도 기록 시점 스냅샷 → 확정 프로젝트 원가 불변. 특별단가·최근 매입가 참조는 **신규 문서 작성 시 기본 제시값**(수정 가능)일 뿐 소급 적용되지 않음.

**예외 경로(지시 반영):** 금액 오류로 세금계산서를 재발행(수정계산서)한 경우에만 원가가 변동된다 — 계산서가 발행된 입고 건의 원가는 "발행 계산서 합계(당초+수정분)"를 따르므로, 수정계산서 발행 즉시 프로젝트 자재비·손익에 자동 반영. 그 외 경로의 원가 변경은 불가. (공용재고 평균단가는 입고 기준 유지 — 정정 필요 시 재고실사/조정 경로 사용, 격차 제안서 4번)

## DC 협상 2모드 (지시 반영)

**총액 DC vs 품번별 DC:** 구매 DC 요청 시 ①총액 DC율(전 품번에 균등 비례 1/N 배분) ②품번별 DC율 중 선택. 공통 원칙 — **발주서의 품번 단가는 원가격 그대로 유지**하고, 확정된 DC를 적용한 **정산가를 단가 옆에 병기**. 공급가액·프로젝트 수익률·미지급금·원가는 전부 정산가 기준으로 자동 계산되므로 품번별 가격 왜곡 없이 수익률 정확성 보장. 공급사 포털의 마진 미리보기도 모드별 정산가 기준으로 계산. 확정(agreed) 후 입고·발주 라인에 원단가+DC율+정산가가 함께 이월·스냅샷.

## v3 추가분 (4차) — 메일함 v2·권한 3단계·입고 모니터링

**조사 근거:** Gmail(별표·일괄 액션바·임시보관함·RE:/FW: 규칙·검색), 네이버웍스(조직 주소록 수신자·안읽음 굵게·수신확인 "읽음 2/3"·발송취소), Odoo 접근권한(앱별 User&lt;Manager&lt;Administrator 3레벨), 이카운트 권한그룹 상속, Monitor ERP(발주확인 미회신 독촉 리스트·부족자재 리스트).

1. **메일함 v2**: 폴더 5종(받은/보낸/임시/보관/휴지통) + 별표·안읽음 필터·검색 + 체크박스 일괄(읽음/안읽음/보관/삭제) + 메일 쓰기(사내 주소록 선택=즉시 전달+수신확인 읽음표시 / 외부=mailto+기록) + 임시저장 + 답장 RE:/전달 FW: 자동 + 미리보기·안읽음 굵게. 삭제는 휴지통 경유(영구삭제 분리). 결재함은 별도 메뉴 유지(지시 반영).
2. **권한 3단계**: 관리자(설정·계정·결재설정+승인) / 책임자(결재 승인/반려) / 사용자(작성·상신). 설정 화면에 권한 매트릭스 표기. 테스트 계정: kim(관리자)/lee2(책임자)/park(사용자), 비밀번호 1234.
3. **발주품 입고 모니터링(프로젝트)**: **실물(수량)** — 미입고(업체·D-day·공급사 준비상태·독촉 버튼)와 입고 완료분 분리 / **전산(금액)** — 발주액(정산가)·입고액(명세서)·계산서 발행액·계산서 미발행·미입고 예정액 분리(지시 반영: 금액적인 부분과 실물 입고를 따로 조회). 미입고 독촉 = ①독촉 메일(미입고 내역 자동 작성) ②포털 긴급표시(같은 시스템 쓰는 업체가 공급사 화면에서 직접 확인). PRJ 카드에 미입고 배지.

4. **납품 전달 → 승인 = 입고 (지시 반영, SAP ASN→GR 승인 패턴)**: 공급사가 확정 명세서에서 [납품 전달] → 발주사 메일함·발주 화면에 승인 요청 도착 → 발주사가 품목·수량 확인 후 **[승인] 시 자동 입고 처리**(프로젝트 원가·재고 반영, 부분 납품이면 잔량은 공급사 "준비중"으로 복귀) / [반려] 시 사유가 공급사 포털에 표시되고 명세서가 수정 가능 상태로 복귀. 상태: prep→final→requested→received.

5. **공정 플랜 vs 실적 (지시 반영 — TECHS 이나즈마선·Odoo 마일스톤·베이스라인 변차 축소판)**: 프로젝트에 설계/발주/자재 입고/조립(+커스텀 공정) 계획 기간을 입력. 실적은 **발주=발주서 일자 자동, 자재 입고=입고 전표(거래명세서 승인 흐름 포함) 자동 연계**, 설계·조립=직접 명기. 타임라인 바(계획=파랑, 실적=초록/지연=빨강, 점선=진행중, 오늘선·납기선)와 공정별 지연 배지(+N일 지연=당겨야 함/제때/−N일 단축)로 "얼마나 밀렸는지" 즉시 확인. PRJ 카드에 최대 지연 배지, 상단에 "최대 +N일 지연 — 일정 조정 검토" 경보.

6. **가격 조율 적용 범위 (지시 반영)**: DC 협상 요청 시 **①이번 건 한정 ②종가 변동(향후 납품분 지속 적용)** 중 선택 — 범위가 공급사 포털에 배지로 표시되어 조율 조건에 포함됨. 종가로 확정되면 기존 업체별 단가는 전일자로 종료되고 새 단가(정산가)가 오늘부터 등록되어 이후 견적요청·발주에 자동 적용(출처 '협상 RFQ번호' 기록). 과거 문서·확정 원가는 불변(원가 확정 원칙 유지).
7. **납품처별 단가 분류·비교 (지시 반영)**: 같은 품목을 여러 곳에서 구매하는 경우 — 품목>단가 관리에 "납품처별 매입가 비교" 표(등록 단가 + 최근 실매입가 + 최근 매입일 + 최저가 배지, 이카운트 '이전 구매처 최근단가' 패턴 확장). 발주 작성 중 품목 선택 시 더 저렴한 납품처가 있으면 💡힌트 토스트.

## v3 추가분 (5차) — 영업/PRJ 분리·메일 설정·주소록·검색

1. **영업/프로젝트 분리 (지시 반영)**: 영업 메뉴 = 대외 문서(견적서·거래명세서·세금계산서)만. 수주(계약)와 공정·입고 모니터링·호기·원가·일정 = PRJ 관리로 이관(프로젝트 목록 표 포함, 기존 링크는 자동 연결).
2. **메일 설정 탭**: 발신 정보(계정 이메일 연동), 사용자별 서명(메일 쓰기 자동 첨부), 외부 메일 계정(IMAP/SMTP) 연동 자리 — Gmail 설정 패턴.
3. **이메일 주소록**: 직접 등록·수정·삭제 + 발송 시 자동 학습(중복 방지) + 메일 쓰기 자동완성(주소록+거래처 용도별 이메일+거래처 담당자 통합) — 네이버웍스 주소록 패턴.
4. **메일 검색 강화**: 필드별 검색(전체/제목/보낸사람/받는사람/본문) + "전체 폴더에서" 검색(결과에 폴더 배지) — Gmail 검색 연산자의 소형판.

# ERP 개발 로드맵 및 v2 상세 설계서

**작성일:** 2026-07-02 · **대상:** 반도체 장비 제작사(5인 이하, 개별 수주생산) 전용 ERP

## 버전 로드맵

| 버전 | 정의 | 상태 |
|---|---|---|
| **v1** | 기본 ERP — 7개 메뉴: 견적→수주(수금계획)→납품→세금계산서, 발주→입고, 통장대사, 조회/장부 | 완료 · /v1/ 보존 |
| **v2** | ①이메일 적용 ②상사 결재(승인) 시스템 ③견적부터 자재·사용처리까지 이력 관리 + 품목 브랜드 | 완료 (erp_v2.html 보존, 검증 160건 통과) |
| **v3 (확정)** | 협업 루프 전체(RFQ→DC 협상 2모드·종가→발주→임시/확정 명세서→납품 승인=입고+첨부), PRJ 관리(공정 플랜vs실적·입고 모니터링 실물/금액 분리·호기/A/S·공수/인력비·도면 링크), 메일함(주소록·서명·검색·수신확인)+결재함, 권한 3단계, 달력/데드라인, 단가 체계(업체별/시기별·납품처 비교·추천), 장부(현금/전산/KPI/카드), 검사성적서·재고실사·마감잠금+감사로그·수출 CI/PL·결제수단, 본사 설정(인증서·홈택스·통장 자리), 백업(스냅샷+복원), 모바일 CSS — **격차 제안 상5+중4 포함 전부 반영** | ✅ 확정 (erp_v3.html 스냅샷 · 자동 검증 573건+상호작용/표 전수 감사 통과) |
| **v4** | v3 운용 피드백 후 결정. 후보: Flask 이행(포털 토큰 URL 실통신·로그인 보안·백업 서버화·SMTP/IMAP), 도면 파일 업로드·리비전 플래그, 호기 단위 발주 태깅, BOM 소요 전개(LRP), 알림톡 승인 | 미정 |

**운영 3원칙 (지시 반영):** ① 데이터 백업이 최우선 — 수동 JSON + 일일 자동 스냅샷(7일) + Flask 이행 시 서버 일일 백업 ② 메뉴 간 상호 연결 — 모든 문서에 "연결 문서" 왕복 체인 ③ 데이터의 시각화 우선 — 진행바·게이지·상태 배지 중심 화면.

원칙: **기본 베이스에 하나씩 붙인다.** 각 버전은 이전 버전을 깨지 않고(데이터 구조 호환), 기능별로 켜고 끌 수 있게 설계한다.

## v2 설계 — 3개 기능의 로직

### A. 상사 결재 (승인 시스템)

**채택 모델** (글로벌 분석 §2): 영림원 게이트형 + Monitor/Odoo 예외형 결합.

**결재 대상 문서 (v2):** 견적서(대외 제출), 발주서(지출). 수주·납품·계산서는 v2에서 비대상 — 중국식 전건 결재는 5인 규모 과잉. 필요해지면 v3에서 문서 타입만 추가하면 되는 구조로 구현.

**상태 기계:**
```
none(미상신) → pending(상신) → approved(승인) 또는 rejected(반려)
                                      반려 → 수정 후 재상신(→pending)
승인된 문서를 수정하면 → none 으로 자동 리셋(재상신 필요)
```

**게이트 규칙 (승인 전 차단되는 행위):**
- 견적서: 이메일 발송, 발송처리, 수주 전환 ← 승인 필요 (설정으로 OFF 가능)
- 발주서: 입고 처리, 이메일 발송 ← 승인 필요
- **예외(자동 통과):** 발주 공급가액 ≤ 승인 한도(기본 100만원, 설정 가능)이면 저장 시 자동 승인. 근거: 미국 실무 원칙 "경영진 승인 건이 전체의 20%를 넘으면 한도를 올려라."

**사용자 모델:** users(대표=승인권한 approver / 직원=staff) + 화면 좌측 사용자 전환(테스트용 — 실개발 시 로그인으로 대체). 승인·상신 기록에 누가/언제/의견 저장.

**결재함 화면:** ①내가 승인할 문서(승인권자만): 문서 요약+금액 카드 → [승인]/[반려+의견] ②내가 올린 문서: 상태 추적 ③결재 설정(한도 금액, 견적 결재 필수 여부). 대시보드에 "결재 대기 N건" 카드.

**v3 확장 자리:** 메신저 카드 승인(알림톡/카카오워크 — 중국 딩톡 패턴, "승인자는 ERP에 로그인하지 않는다"), 다단계 결재선, 문서 타입 추가.

### B. 이메일 연동

**무료 구간 (v2에서 완전 구현):**
1. 거래처 마스터의 이메일 필드 활용(v1 기존).
2. 견적서·발주서·거래명세서에 **[이메일 발송]** 버튼 → 제목("[회사명] 견적서 Q2026-0001")·본문(품목/금액/납기 요약)을 자동 생성 → 수정 가능 → **mailto:로 기본 메일 앱 호출**(서버·비용 불요).
3. **발송 기록(타임라인):** 문서별로 언제/누구에게/무슨 제목으로 보냈는지 저장, 문서 상세와 발송 모달에서 이력 확인 (Odoo 채터 패턴).
4. 결재 게이트: 미승인 문서는 발송 버튼 차단.

**유료/서버 구간 (자리만 두고 v2에서는 미구현):**
- `send_email_smtp()` — Flask 이행 시 SMTP(예: 구글 워크스페이스)로 자동 발송+PDF 첨부. mailto 방식의 한계(첨부 불가, 발송 여부 미확인)를 이 단계에서 해소.
- 수신 열람 추적, 카카오 알림톡, 발주확인 미회신 독촉 리스트(Monitor 패턴).

### C. 견적→발주→입고→자재→사용처리 이력 (프로젝트 전체 단가)

**원가 귀속 2경로 (미국 잡샵 표준):**
```
경로1 Buy-to-job: 수주 연결 발주 → 입고 확인 = 즉시 그 수주의 자재비 (v1 기존)
경로2 공용재고:   공통 발주(수주 미연결) → 입고 = 자재 재고 적립
                → [사용처리(불출)] 수주 선택 → 불출 시점에 그 수주의 자재비
                → '기타(소모/샘플)' 불출은 원가 아닌 소모 기록
```

**로직 상세:**
- 재고 수량 = Σ공통입고 − Σ사용처리 (저장하지 않고 계산 — v1 원칙 유지)
- 불출 단가 = 공통입고의 **이동평균 매입단가** 스냅샷 (영림원 월마감 단가계산은 과잉 → 즉시 인식으로 단순화. 프로젝트 직송분은 개별법 그대로)
- 재고 초과 불출 차단, 불출 취소 시 재고 복구
- 수주 화면의 원가 패널 = **직납 자재비 + 불출 자재비** 분리 표시 + [자재 투입 내역] 전체 이력(발주→입고 건별 + 불출 건별)
- **이력 체인 완성:** 견적 → 수주 → 발주 → 입고 → (자재재고) → 사용처리 → 수주 원가, 모든 문서가 "연결 문서" 패널로 왕복 — 어느 화면에서든 전체 사슬 추적

**v3 확장 자리:** BOM(자재명세) 등록 → 수주 1건 소요 전개(易飞 LRP), 부족자재 리스트(Monitor), 공용재고 프로젝트 간 대체 전용 화면(영림원 반도체 패키지), 호기별(장비 1대별) 원가.

## 3. v2 개발 순서 (의존성 순)

1. 사용자·설정 (users, 승인한도, 사용자 전환) — 다른 기능의 전제
2. 결재 데이터·상태 기계 (상신/승인/반려/자동통과/수정 시 리셋)
3. 게이트 적용 (견적: 발송·전환 / 발주: 입고·발송)
4. 결재함 화면 + 대시보드 카드
5. 자재 재고 계산 (공통 입고 − 불출) + 자재 탭
6. 사용처리(불출) + 수주 원가 합산 변경 + 원가 구성·투입 이력 화면
7. 이메일 모달 (자동 본문 → mailto → 발송 기록) + 게이트 연동
8. 샘플 데이터 갱신 + 자동 테스트(기존 159건 + 신규 시나리오) 통과

## 4. 실개발(Flask) 이행 시 스키마 반영 (v2 기준 추가분)

`users`, `approvals`(doc_type, doc_id, status, requested_by/at, decided_by/at, comment), `email_logs`(doc_type, doc_id, to, subject, body, method, sent_by/at), `material_issues`(item_id, qty, unit_cost, sales_order_id nullable, note, issued_by/at), `settings`(approval_limit, quote_approval_required, smtp_config). 총 15+5=20테이블. 기존 P1 지적사항(정수 금액, 배타적 FK CHECK, 채번, CSV 해시, 수정계산서 자기참조, created_by, 과세혼합)과 함께 schema v3로 확정 권장.

## 5. v3 후보 목록 (v2 검증 후 선택)

우선순위 논의용 — 반도체 장비사 관점 추천순:
1. **수주별 원가 예산(견적원가) → 예산 vs 실적 vs 예측 3값** (TECHS 패턴 — "이 장비 남나 밑지나" 조기 경보)
2. **BOM + 수주 1건 소요 전개(LRP)** → 부족자재 리스트 → 발주 자동 생성
3. **호기별(장비 Unit별) 관리** — 1수주 다장비의 호기별 원가·이력 (영림원 반도체 패키지)
4. **A/S·설치 이력 관리** + 대표수주번호(본계약+추가작업 묶음 손익, TECHS)
5. **메신저 승인·알림** (알림톡/카카오워크 카드 승인)
6. **SMTP 자동 발송 + PDF 출력 양식** (견적서·거래명세서 인쇄물)
7. 홈택스·오픈뱅킹 연동 (유료 — 기존 설계의 자리)
8. 은행 대사 자동 규칙 (e-conomic 패턴 — "급여" 적요 자동 기타처리)
9. 로그인·권한, 일일 백업 자동화 (Flask 이행과 동시)

---

**v2 완료 기준:** 자동 테스트 전건 통과 + 다음 시나리오가 클릭만으로 재현될 것 — "직원이 견적 작성→상신, 대표가 결재함에서 승인, 견적 이메일 발송(기록 남음), 수주 전환, 고액 발주 상신→승인→입고, 공통 자재 입고→수주로 불출, 수주 화면에서 직납+불출 합산 원가와 투입 이력 확인."

## 유료·가입 필요 항목 명기 (지시 반영 — 비용 경계표)

| 항목 | 필요 시점 | 비용/가입 |
|---|---|---|
| 홈택스 전자세금계산서 전송 | 실서비스 | 중계사(팝빌/바로빌) **가입 + 건당 과금**(약 100~200원/건), 공동인증서 필요 |
| 오픈뱅킹/계좌 자동수집 | 실서비스 | 스크래핑(쿠콘 등) **유료 계약** 또는 오픈뱅킹(핀테크 등록) — 현재 CSV 무료 대체 가동 |
| 카드 사용내역 자동수집 | 실서비스 | 카드사 API/여신협회 **가입·유료** — 현재 CSV 무료 대체 가동 |
| SMS/알림톡 | 선택 | 발신 프로필 **가입 + 건당 과금**(알림톡 약 7~10원) |
| SMTP 자동 메일 발송 | 실서비스 | 무료 티어 가능(Brevo 300통/일 등 **가입**), 구글 워크스페이스는 유료 |
| 외부 메일 수신(IMAP/Gmail API) | 실서비스 | **무료**(가입만) — 서버 상시 구동 필요 |
| OCR(영수증/명세서 인식) | 선택 | 클로바 OCR 등 **유료 API + 가입** |
| LLM 문구 생성 | 선택 | API **유료 + 가입** — 현재 규칙 기반 자동 본문 무료 대체 |
| 공동인증서(사업자범용) | 실서비스 | 발급 **유료**(연 10만원 내외) |
| 서버/도메인 (Flask 호스팅) | 실서비스 | 월 1~3만원 수준(경량 VPS) + 도메인 연 2만원 내외 |
| GitHub Pages 테스트 배포 | 현재 | **무료**(가입만) |

무료 대체 경로가 시스템에 이미 내장된 항목: 계산서 내부기록 발행, 통장·카드 CSV 업로드, mailto 발송, 인쇄→PDF.

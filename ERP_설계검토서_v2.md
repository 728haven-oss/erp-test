# 장비 제조사 ERP 설계서(v2) 검토 보고서

**검토일:** 2026-07-02 · **대상:** 7개 메뉴 상세 설계서 v2 (Flask, SQLite→PostgreSQL, 15개 테이블)

## 총평

방향이 옳습니다. 헤더+라인 분리, 품명 스냅샷, 미수금 저장 없이 계산, 삭제 대신 비활성, 상태 기반 수정 통제, M:N 입금 배분 — 모두 소규모 수주생산 ERP에 적정한 선택입니다. 특히 "회차별 수금계획 대비 실입금" 구조(A-3)를 핵심으로 잡은 것이 이 설계의 강점입니다.

아래는 우선순위별 보완 사항입니다. P1은 스키마 확정 전에 반드시 결정해야 하는 것들로, 나중에 바꾸면 데이터 마이그레이션이 필요합니다.

## P1 — 스키마 확정 전 필수 결정 (7건)

### 1. 금액 자료형과 부가세 반올림 규칙
SQLite에는 DECIMAL이 없어 REAL(부동소수점)로 저장하면 원 단위 오차가 생깁니다. **금액은 전부 INTEGER(원 단위)로 저장**하세요. 부가세 반올림 기준(라인별 10% 반올림 후 합산 vs 공급가 합계의 10%)도 지금 정해야 합니다. 두 방식은 결과가 1~2원 다를 수 있고, 세금계산서의 라인 세액 합 = 헤더 세액이 어긋나면 홈택스 전송 시 반려됩니다. 권장: 라인별 반올림 후 합산(홈택스 표준 방식).

### 2. 배타적 외래키 CHECK 제약
설계상 "N택1" FK가 3곳 있는데, CHECK 없이 nullable로만 두면 둘 다 채워지거나 둘 다 비는 데이터가 생깁니다.

- `tax_invoices`: sales_order_id / transaction_id / installment_id 중 **정확히 1개**
- `transactions`: sales_order_id(납품) / purchase_order_id(입고) 중 정확히 1개 + direction 컬럼
- `payment_allocations`: installment_id(입금) / purchase_transaction_id(출금) 중 정확히 1개

예: `CHECK ((sales_order_id IS NOT NULL) + (transaction_id IS NOT NULL) + (installment_id IS NOT NULL) = 1)`

### 3. 문서번호 채번 규칙
견적·수주·명세서·계산서 모두 사람이 부르는 번호가 필요한데 설계에 없습니다. `Q2026-0001` 식 연도별 시퀀스를 권장하고, 채번 테이블 또는 `SELECT MAX` + 트랜잭션으로 중복을 막으세요.

### 4. 통장 CSV 재업로드 중복 방지
같은 CSV를 두 번 올리거나 기간이 겹치게 올리는 일은 반드시 생깁니다. `bank_transactions`에 (계좌, 거래일시, 금액, 적요) 해시 UNIQUE를 걸어 중복 행을 자동 스킵하세요. "원본 불변" 원칙의 전제 조건입니다.

### 5. 수정세금계산서 자기참조
`tax_invoices.original_invoice_id`(자기참조 FK)와 수정사유 코드 컬럼을 지금 넣어두세요. 지금은 내부기록이라 취소 후 재발행으로 단순화해도 되지만, 홈택스 연동 시 수정사유 코드(기재정정·착오 등 6종)가 필수라 컬럼 자리는 미리 필요합니다.

### 6. 사용자 식별 (users 테이블 추가 → 16개)
"여러 PC 공유"인데 계정 개념이 없습니다. 간단 로그인 + 전 테이블 `created_by`/`updated_by`만 있어도 "이 수주 금액 누가 바꿨나"에 답할 수 있습니다. 권한 체계는 불필요하고 식별만 하면 됩니다.

### 7. 과세/면세 혼합 문서 규칙
품목에 과세/면세 구분이 있으므로 한 수주에 혼합될 수 있습니다. 세금계산서(과세)와 계산서(면세)는 별도 문서이므로, 발행 시 과세분/면세분을 분리 발행하는 규칙을 정하거나, v1에서는 "혼합 문서 발행 시 경고 후 수동 분리"로 단순화하세요.

## P2 — 개발 중 반영 (6건)

1. **동시 수정 충돌 감지**: 여러 PC 공유이므로 저장 시 `updated_at` 비교(낙관적 잠금)로 "다른 사람이 먼저 수정했습니다" 경고. `updated_at`을 이미 전 테이블에 두기로 했으니 비용이 거의 없습니다.
2. **부분 납품 → 수주 상태 판정 기준**: 라인별 누적 납품수량 = 수주수량일 때 delivered 처리인지, 수동 전환인지 명시 필요. 초과 납품 경고도.
3. **수주 취소 흐름 정의**: canceled 잠금은 있는데, 이미 발행된 계산서·배분된 입금이 있을 때의 절차(배분 취소 → 계산서 취소 → 수주 취소 순서 강제)가 없습니다.
4. **매입 외 출금 처리**: 통장 출금에는 급여·경비·이체 등 매입 아닌 건이 다수입니다. '기타' 분류로 대사 완료 처리할 수 있어야 미대사 건수가 0으로 수렴합니다.
5. **미지급금 현황**: 매입 계산서 수취 → 미지급금 발생까지는 있는데 조회 화면이 없습니다. 조회/장부 메뉴에 "거래처별 미지급 현황" 추가.
6. **배분 합계 검증**: 한 통장 건의 배분 합계 ≤ 원금을 저장 시점에 검증(앱 레벨) + 일일 정합성 점검 쿼리.

## P3 — 선택 (여유 있을 때)

수주 첨부파일(도면·계약서 스캔), 견적서 rev 관리(수정 요청 시 복사 후 개정), 원가 범위 명시(자재비만, 노무비·경비 제외임을 화면에 표기), 변경 이력 테이블(소규모면 updated_by로 충분할 수 있음).

## 운영·인프라 (별도 강조)

- **SQLite 다중 PC의 올바른 구조**: Flask 서버 1대(사무실 PC 상시 구동)에 DB를 두고 나머지는 브라우저로 접속. **NAS 공유폴더에 .db 파일을 두고 여러 PC가 직접 여는 구조는 파손 위험이 커서 금지.** WAL 모드 활성화. 이 전제를 설계서에 명시하세요.
- **PostgreSQL 이행 대비**: SQLAlchemy(또는 최소한 DB 추상화 계층) 사용, SQLite 전용 문법 회피. 그래야 "SQLite→PostgreSQL"이 설정 변경 수준이 됩니다.
- **백업**: 소기업 ERP 최대 리스크는 기능 부족이 아니라 데이터 유실입니다. 일일 자동 백업(로컬+외부 1곳) + 분기 1회 복구 테스트를 운영 요건으로 넣으세요.

## 결론

P1 7건을 반영해 schema_v2 → v3로 확정한 뒤 개발 착수를 권장합니다. 메뉴 구성·상태 흐름·만드는 순서(D)는 그대로 진행해도 좋습니다.
